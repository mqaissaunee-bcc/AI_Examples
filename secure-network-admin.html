Common Network Protocols & Ports

FTP • SSH • Telnet • SMTP • DNS • HTTP/S • LDAP • SNMP • RDP

Why Protocols & Ports Matter for Security

Every network service communicates over a defined protocol and port. Understanding these mappings allows security professionals to identify misconfigurations, unauthorized services, and attack vectors. The SY0-701 exam requires you to not only memorize port numbers but to understand the security implications — specifically the difference between protocols that transmit data in plaintext versus those that provide encryption.

Key Principle

Plaintext protocols are a security risk. Protocols like Telnet, FTP, HTTP, SNMP v1/v2, and POP3 transmit credentials and data without encryption — anyone on the network who can capture packets can read usernames, passwords, and content. Always prefer the encrypted equivalent.

Protocols by OSI Layer

Understanding which OSI layer a protocol operates at helps determine what information is available for security controls and how attacks can be detected or mitigated.

Layer 7 — Application

HTTP/S, FTP/SFTP, SMTP, POP3, IMAP, DNS, DHCP, LDAP, SNMP, Telnet, SSH, RDP, Kerberos, RADIUS, TACACS+

Layer 6 — Presentation

TLS/SSL encryption and decryption occurs here. Handles data format, compression, and encryption translation between application and session layers.

Layer 5 — Session

NetBIOS (ports 137-139), RPC (port 135), SMB (port 445). Session establishment, maintenance, and termination for application communications.

Layer 4 — Transport

TCP (reliable, connection-oriented) and UDP (unreliable, connectionless). Port numbers exist at this layer. Firewalls inspect ports here.

Layer 3 — Network

IP (v4/v6), ICMP, IPsec, OSPF, BGP. Routing and logical addressing. IP spoofing and routing attacks occur here.

Layer 2 — Data Link

ARP, MAC addresses, 802.1X port security, VLANs, STP. Switch-level attacks (ARP spoofing, VLAN hopping) target this layer.

Exam Tip

The exam frequently asks you to identify what layer a firewall, IDS, or protocol operates at. Packet-filtering firewalls = Layer 3-4; stateful = Layer 4; application/proxy firewalls = Layer 7. SSL/TLS inspection happens at Layer 6-7. Understanding this helps place controls correctly in network architecture scenarios.

Secure Protocol Implementations

TLS 1.3 • SSH • HTTPS • SRTP • DNSSEC • DoH • DoT

The Principle: Encrypt Everything in Transit

Modern network security demands that all data in transit be encrypted. When an insecure protocol cannot be replaced, a secure transport layer should wrap it (e.g., TLS tunnelingTLS tunnelUsing TLS to encapsulate otherwise insecure protocols, providing encryption for data in transit regardless of the application-layer protocol.). The shift from plaintext to encrypted communication is one of the most important security improvements an organization can make.

TLS (Transport Layer Security)

TLSTLSTransport Layer Security — the cryptographic protocol that secures communications over networks. TLS 1.3 is the current standard, dropping deprecated cipher suites and improving the latency of the handshake. is the foundation of secure communications on the internet. It provides:

  • Confidentiality — data is encrypted in transit
  • Integrity — MAC (Message Authentication Code) detects tampering
  • Authentication — certificates verify server (and optionally client) identity

TLS 1.3 (Current Standard — 2018)

  • Removed weak cipher suites (RC4, 3DES, SHA-1)
  • Only supports PFSPerfect Forward Secrecy (PFS)A property ensuring that session keys are not compromised even if long-term secrets are compromised. Uses ephemeral key exchange (ECDHE) so each session has a unique key. cipher suites (ECDHE)
  • Reduced handshake to 1-RTT (0-RTT for session resumption)
  • Mandatory: AES-GCM, ChaCha20-Poly1305
  • Removes static RSA key exchange

Deprecated Versions (Do Not Use)

  • SSL 2.0 / SSL 3.0 — completely broken (POODLE, DROWN attacks)
  • TLS 1.0 — vulnerable to BEAST attack; deprecated per RFC 8996 (2021)
  • TLS 1.1 — deprecated; limited cipher support
  • TLS 1.2 — still acceptable if misconfigured options removed; prefer 1.3
Common Attack: Downgrade

Attackers attempt to force clients and servers to negotiate a weaker version of TLS or revert to SSL. Mitigate with: enforce minimum TLS 1.2 on servers, implement HSTS (HTTP Strict Transport Security) for web servers, and use SSL/TLS inspection to verify that the TLS version being negotiated is acceptable.

SSH (Secure Shell)

SSH is the cornerstone of secure remote administration. It replaces Telnet, rlogin, rsh, and rcp with encrypted equivalents. SSH provides:

  • Encrypted terminal/shell access (replaces Telnet)
  • Secure file transfer via SFTP (SSH File Transfer Protocol)
  • Secure copy via SCP (Secure Copy Protocol)
  • Port forwarding / tunneling for other protocols
  • Key-based authentication (far more secure than passwords)

SSH Key-Based Authentication

Generate an asymmetric key pair. Public key stored on server (~/.ssh/authorized_keys). Private key stays with user. No password transmitted — immune to password-based brute force.

SSH Hardening

Disable root login (PermitRootLogin no), disable password auth (PasswordAuthentication no), restrict allowed users/groups, use SSH v2 only, change default port (security through obscurity — not a primary control), implement fail2ban or equivalent.

SSH Tunneling / Port Forwarding

SSH can create encrypted tunnels for other protocols. Local forwarding forwards traffic from local port through SSH to remote host. Be aware: attackers can use SSH tunneling to bypass firewall controls and exfiltrate data.

DNSSEC, DNS over HTTPS (DoH), and DNS over TLS (DoT)

Standard DNS (port 53) transmits queries and responses in plaintext — anyone on the network path can see and potentially manipulate DNS resolution. Three security extensions address this:

DNSSECDNSSECDNS Security Extensions — adds digital signatures to DNS records. Does NOT encrypt DNS — it provides authentication and integrity. A resolver that validates DNSSEC can verify records are authentic. digitally signs DNS records using asymmetric cryptography. When a resolver queries a DNSSEC-signed zone, it can verify the response was not tampered with. Key record types: DNSKEY, RRSIG, DS, NSEC/NSEC3. Important: DNSSEC provides integrity and authentication — it does NOT encrypt DNS queries or hide what you are querying.

DNS over HTTPS encrypts DNS queries inside HTTPS traffic (port 443). Benefits: privacy (ISPs/network admins cannot see DNS queries), resistance to interception and manipulation. Challenge: queries blend with web traffic — makes network monitoring more difficult. Supported by most modern browsers and OSes. Organizations should consider whether to allow or proxy DoH.

DNS over TLS encrypts DNS queries using TLS on a dedicated port (853). Similar privacy benefits to DoH but uses a separate port that is identifiable on the network — easier for organizations to allow or block. Preferred for enterprise environments where network administrators want to inspect or control DNS traffic flow while still providing encryption.

SRTP, S/MIME, and LDAPS

📞 SRTP — Secure RTP

Secures VoIP and real-time media streams. Adds encryption (AES), authentication, and replay protection to RTPRTPReal-Time Protocol — transports audio and video over IP networks. Used by VoIP systems. Without SRTP, voice calls can be intercepted and decoded.. Use SRTP with DTLS for key exchange in modern VoIP implementations (WebRTC, SIP). Always use SRTP — unencrypted VoIP is a significant eavesdropping risk.

✉ S/MIME — Secure Email

S/MIME (Secure/Multipurpose Internet Mail Extensions) provides email encryption and digital signatures. Uses certificates to sign and/or encrypt email content. Unlike TLS (which protects the transmission), S/MIME protects the message itself — even when stored or forwarded. Requires certificate exchange between sender and recipient.

👤 LDAPS (LDAP Secure)

LDAP over TLS (port 636). Required for secure Active Directory authentication. Prevents credential interception during directory lookups. Organizations should enforce LDAP channel binding and LDAP signing on domain controllers. Without LDAPS, credentials are exposed during every domain authentication.

🎶 SFTP vs FTPS

SFTP = SSH File Transfer Protocol (port 22) — completely separate protocol from FTP; runs over SSH. FTPS = FTP + TLS (port 990 implicit, or 21 explicit with STARTTLS). Both provide encrypted file transfer. SFTP is generally preferred — simpler firewall configuration (single port), better cross-platform support.

SPF (Sender Policy Framework): A DNS TXT record listing authorized mail servers for a domain. Receiving mail servers check SPF to verify the sending server is allowed to send email for the sender's domain. Helps prevent email spoofing of the envelope sender. Does not protect the header "From" address.
DKIM (DomainKeys Identified Mail): Adds a digital signature to outgoing email headers. The private key is held by the sending mail server; the public key is published in DNS. Receiving servers verify the signature against the DNS record. Protects email integrity — confirms the email was not modified in transit.
DMARC (Domain-based Message Authentication, Reporting, and Conformance): Builds on SPF and DKIM. A DNS policy that tells receiving servers what to do when email fails SPF and DKIM checks: none (report only), quarantine (spam folder), or reject (refuse delivery). DMARC also enables aggregate reporting so senders can monitor email authentication results.
S/MIME: Provides end-to-end encryption and digital signatures for email message content. Protects the message even at rest and when forwarded. Requires that both sender and recipient have valid certificates and have exchanged public keys. Supported by most enterprise email clients (Outlook, Thunderbird, Apple Mail).
STARTTLS / Opportunistic TLS: Upgrades a plaintext SMTP connection to TLS encryption. Most modern SMTP servers support STARTTLS. However, opportunistic TLS is vulnerable to downgrade attacks — an attacker can strip the STARTTLS command. Use MTA-STS (Mail Transfer Agent Strict Transport Security) to enforce TLS between mail servers.

Remote Access Technologies

VPN • IPsec • SSL VPN • RDP • Jump Server • Out-of-Band Management

Securing Remote Administration

Remote access is one of the highest-risk surfaces in enterprise security. Remote access provides the same capabilities as on-site access, but over untrusted networks — making robust authentication, encryption, and monitoring essential. High-profile breaches have repeatedly been traced to compromised remote access credentials (especially RDP).

VPN Technologies

IPsec VPN

Network-layer encryption (Layer 3). Two modes: Transport (encrypts payload only — used host-to-host) and Tunnel (encrypts entire IP packet — used for site-to-site and remote access). Uses IKEv2 for key exchange. Supports AH (Authentication Header) and ESP (Encapsulating Security Payload). Most secure VPN option.

SSL/TLS VPN

Operates at Layer 4-7. Accessed via browser (clientless) or thin client. Uses TCP port 443 — typically allowed through firewalls. Easier to deploy than IPsec. Less overhead for remote workers. Examples: OpenVPN (UDP 1194 or TCP 443), Cisco AnyConnect. Vulnerable to web-based attacks if not hardened.

Split vs. Full Tunnel

Full tunnel: All traffic routed through VPN — better security, higher bandwidth overhead. Split tunnel: Only corporate traffic goes through VPN; internet traffic bypasses it — reduced load but creates a gap where internet traffic is unmonitored. Consider the security implications based on policy.

Always-On VPN

Automatically connects corporate-managed devices to VPN whenever outside the corporate network. Ensures all endpoint traffic is protected regardless of user action. Implemented via MDM/endpoint management. Prevents employees from accidentally connecting to public Wi-Fi without protection.

RDP (Remote Desktop Protocol) Security

RDP (port 3389) allows full graphical remote access to Windows systems. It is one of the most targeted services by attackers, responsible for numerous ransomware and data breach incidents. Never expose RDP directly to the internet.

⚠ RDP Attack Vectors

  • BlueKeep (CVE-2019-0708) — unauthenticated RCE via RDP (pre-auth)
  • Brute force — automated password guessing on port 3389
  • Credential stuffing — using breached passwords
  • Pass-the-Hash — using captured NTLM hashes for RDP authentication
  • RDP MITM — downgrade attacks if NLA disabled

🛡 RDP Hardening Controls

  • Never expose port 3389 to the internet
  • Require VPN + MFA before RDP access
  • Enable NLA (Network Level Authentication)
  • Implement account lockout policies
  • Restrict RDP to specific IP ranges (firewall ACLs)
  • Apply all Microsoft security patches promptly
  • Monitor RDP access in SIEM; alert on unusual login times or locations

Jump Servers / Bastion Hosts

A jump serverJump server (bastion host)A hardened intermediary server that serves as the sole entry point for remote administrative access to a network. All administrative connections must pass through the jump server, creating a single choke point for auditing and access control. (also called a bastion host or privileged access workstation) is a hardened intermediary that provides the only path for administrative access to sensitive systems.

How It Works

Administrators connect to the jump server first (via VPN + MFA), then initiate connections from the jump server to internal resources. The jump server is the only host with network access to management interfaces. Eliminates direct administrative access from user workstations.

Security Benefits

Centralized logging of all administrative sessions, single point for MFA enforcement, reduced attack surface (admin credentials not stored on user workstations), ability to record and audit all privileged commands, and simplified firewall rules (only jump server needs access to admin ports).

Jump Server Hardening

The jump server itself must be hardened: patch aggressively, minimal services, dedicated for admin use only, strong MFA, session recording, host-based IDS, no internet access, regular vulnerability scanning, and privileged access workstation (PAW) policy.

Out-of-Band (OOB) Management

Out-of-band management provides a separate network path for administrative access to network devices, completely independent of the production network. Critical for incident response — allows administrators to access and reconfigure systems even when the primary network is compromised or unavailable.

A dedicated management network (separate VLAN or physical network) connects to management interfaces of all network devices (routers, switches, servers). This network carries only management traffic — no user data. Compromise of the production network does not affect management access. Typically uses a dedicated management switch and separate routing infrastructure.

Server hardware includes dedicated management controllers: IPMI (Intelligent Platform Management Interface), iDRAC (Dell), iLO (HP), and BMC (Baseboard Management Controller). These provide out-of-band access to power cycle, reimage, and console access to servers even if the OS is unresponsive. Security note: IPMI has had critical vulnerabilities — always isolate management interfaces on a dedicated OOB network and apply firmware updates.

Console servers (terminal servers) provide remote access to serial console ports of network equipment. Even if the network interface of a router or switch is misconfigured, administrators can connect via the console server to correct the issue. Typically access is secured with SSH + authentication to the console server, which then provides serial console access to equipment.

AAA Frameworks & 802.1X Port Security

RADIUS • TACACS+ • Kerberos • EAP • 802.1X • NAC

Authentication, Authorization, and Accounting (AAA)

AAA is the framework for controlling who can access network resources, what they are permitted to do, and maintaining an audit trail of their activities. Understanding AAA protocols is critical for designing secure network access — and for correctly answering scenario-based Security+ questions about network authentication.

🔑 Authentication

Verifies identity — "Who are you?" Answers this question by validating credentials (something you know, have, or are). Examples: username/password, certificates, biometrics, hardware tokens, and MFA combinations.

🔒 Authorization

Determines access rights — "What are you allowed to do?" After authentication, the system evaluates the subject's permissions. Can be based on roles (RBAC), attributes (ABAC), or mandatory labels (MAC).

📝 Accounting

Records activity — "What did you do?" Logs session details, commands executed, data accessed, time, source IP, and duration. Critical for forensic investigations, compliance, and detecting anomalous behavior.

RADIUS vs. TACACS+

RADIUS

  • Open standard (RFC 2865)
  • UDP transport (ports 1812/1813)
  • Encrypts password only — other attributes in plaintext
  • Combines Authentication and Authorization in single response
  • Accounting on port 1813
  • Widely supported — wireless, 802.1X, VPN, dial-up
  • Best for network access control (connecting to network)

TACACS+

  • Cisco-developed (not open, but widely implemented)
  • TCP transport (port 49) — reliable delivery
  • Encrypts entire packet body — more secure
  • Separates Authentication, Authorization, Accounting
  • Granular command-level authorization
  • Best for device administration (managing routers, switches)
  • Allows different AAA servers for each function
Exam Tip — RADIUS vs TACACS+

A common exam question: "Which AAA protocol should be used for device administration with granular command-level authorization?" Answer: TACACS+ — because it separates authorization and allows per-command control. For wireless/VPN network access: RADIUS is the more common choice. TACACS+ also encrypts the entire packet while RADIUS only encrypts the password field.

EAP Variants

Requires certificates on both client and server — mutual TLS authentication. Most secure EAP method. Prevents man-in-the-middle attacks. Requires PKI infrastructure to issue client certificates. Used in high-security environments where certificate management overhead is acceptable.

Creates an encrypted TLS tunnel using only a server certificate, then tunnels an inner authentication method (typically MSCHAPv2 or EAP-GTC). More practical than EAP-TLS because clients don't need certificates — they authenticate with credentials inside the tunnel. Widely used in enterprise WPA2/WPA3 Enterprise wireless.

Similar to PEAP — uses a server certificate to establish a TLS tunnel, then tunnels any inner authentication protocol (including PAP, CHAP, MS-CHAP). More flexible than PEAP. Commonly used in environments with diverse authentication methods. Only server certificate required (not client certificate).

Cisco-developed EAP method designed to replace LEAP. Uses Protected Access Credentials (PAC) instead of certificates to establish a TLS tunnel. Faster than certificate-based methods. Suitable for environments without PKI infrastructure.

DNS & DHCP Security

DNS Poisoning • DHCP Starvation • Rogue Servers • DNSSEC • DHCP Snooping

Why DNS and DHCP Are Critical

DNS and DHCP are foundational network services that are often overlooked from a security perspective. DNS translates names to addresses — if compromised, users can be redirected to malicious sites. DHCP assigns IP addresses and gateway information — if compromised, attackers can redirect all traffic through a malicious gateway. Both services operate with minimal authentication, making them high-value attack targets.

DNS Attack Techniques

⚠ DNS Cache Poisoning

Attacker injects false DNS records into a resolver's cache. Victims querying that resolver receive malicious IP addresses for legitimate domain names — redirecting traffic to attacker-controlled servers. Exploits the lack of authentication in standard DNS. Famous example: Kaminsky Attack (2008). Mitigation: DNSSEC, randomize source ports, minimize TTL, use 0x20 encoding.

⚠ DNS Hijacking

Attacker modifies DNS settings at the device, router, or registrar level to redirect queries to a malicious resolver. Can be achieved via malware, compromised home router, or registrar account takeover. Results in silently redirecting all web traffic through attacker-controlled infrastructure.

⚠ DNS Amplification (DDoS)

Attacker sends DNS queries with spoofed source IP (victim's IP) to open resolvers requesting large responses (e.g., ANY record type). Open resolvers send large responses to victim, amplifying traffic 50-100x. Used as a DDoS amplification vector. Mitigation: Disable open resolvers, implement BCP38 (anti-spoofing), rate limiting.

⚠ DNS Tunneling

Encodes data in DNS queries and responses to exfiltrate data or establish C2 channels through firewalls that allow DNS traffic. DNS queries for unusually long subdomains carry encoded data. Detection: monitor for unusually long DNS queries, high query volumes to unusual domains, or domains using randomized subdomains.

⚠ Zone Transfer Abuse

DNS zone transfers (AXFR) are used to replicate DNS data to secondary servers. If allowed from any IP, attackers can retrieve a complete list of all hostnames and IP addresses in a zone — valuable for reconnaissance. Restrict zone transfers to authorized secondary DNS servers only.

⚠ Typosquatting / IDN Homograph

Registering domains with slight variations (typosquatting: googl.com) or using similar-looking Unicode characters (IDN homograph: аpple.com where 'а' is Cyrillic) to trick users. Combine with phishing emails. Mitigated by user education, email filtering, and browser protections.

DNS Security Controls

  • Implement DNSSEC on all authoritative zones — sign records with DNSKEY
  • Restrict zone transfers (AXFR) to authorized secondary DNS servers only
  • Disable recursive resolution on authoritative servers (or restrict to internal resolvers)
  • Use DNS filtering / RPZ (Response Policy Zones) to block malicious domains
  • Deploy DNS over TLS (DoT) or DNS over HTTPS (DoH) for encrypted resolution
  • Monitor DNS logs for long queries, unusual TLD patterns, or DNS tunneling indicators
  • Rate limit DNS responses to prevent amplification attacks
  • Register defensive domain variations to prevent typosquatting impersonation

DHCP Attack Techniques

⚠ DHCP Starvation

Attacker sends large volumes of DHCP DISCOVER messages with spoofed MAC addresses, exhausting the DHCP server's address pool. Legitimate clients cannot receive IP addresses — effectively a Denial of Service against network access. Tool: Yersinia, gobbler.

Mitigation: DHCP snooping + port-based MAC address limiting on switches.

⚠ Rogue DHCP Server

Attacker deploys an unauthorized DHCP server that responds to client DISCOVER messages before the legitimate server. Clients receive malicious gateway/DNS server addresses — redirecting all traffic through attacker infrastructure (classic MITM setup). Tool: Any machine with DHCP server software.

Mitigation: DHCP snooping — trust only specific ports.

Attack Scenario

A rogue DHCP server attack is particularly dangerous: An attacker in a coffee shop or corporate network deploys a DHCP server that answers faster than the legitimate server. Victims receive the attacker's machine as the default gateway — all traffic passes through it. Combined with SSL stripping or a transparent proxy, the attacker can intercept credentials, session tokens, and sensitive data.

DHCP Security Controls

🔌 DHCP Snooping

A Layer 2 switch feature that distinguishes between trusted (server-facing uplinks) and untrusted (client-facing) ports. Only DHCP responses from trusted ports are forwarded. Untrusted ports cannot send DHCP OFFER or ACK messages — rogue DHCP servers on client ports are blocked. Builds a DHCP snooping binding table.

🔌 Dynamic ARP Inspection (DAI)

Uses the DHCP snooping binding table to validate ARP packets on untrusted ports. Prevents ARP poisoning/spoofing attacks that complement rogue DHCP attacks. Checks that ARP replies match the known IP-to-MAC binding from DHCP snooping. Drop invalid ARP packets silently.

🔌 IP Source Guard

Extends DHCP snooping by filtering IP traffic on untrusted ports. Checks that source IP and MAC in packets match the DHCP snooping binding table. Prevents IP spoofing at the switch port level. Works in conjunction with DHCP snooping and DAI for comprehensive Layer 2 security.

📌 DHCP Lease Monitoring

Monitor DHCP logs for unusual patterns: high DISCOVER rate (starvation attempt), unexpected OFFER from unknown server IP/MAC (rogue server), rapid lease exhaustion. Correlate DHCP data with network access logs. Alert on new DHCP servers appearing on the network.

Network Hardening Techniques

Disable Unused Services • SNMPv3 • Secure Baselines • Port Security • VLAN Segmentation

Defense in Depth: Hardening Principles

Network hardening reduces the attack surface by eliminating unnecessary services, strengthening configurations, and applying the principle of least privilege to network access. A hardened network has fewer entry points for attackers and limits the lateral movement available when a breach occurs.

Minimize Attack Surface

Disable every service, port, and protocol not explicitly required. Every enabled service is a potential vulnerability. Every open port is a potential entry point. "Default deny" — block everything not explicitly permitted.

Least Privilege

Users and systems receive only the access they need to perform their function. Admin accounts are separate from user accounts. Network devices have limited management access (only authorized admin IPs, only SSH — no Telnet).

Secure Baselines

Document and enforce a secure configuration baseline for every device type. Use CIS Benchmarks, DISA STIGs, or vendor security guides as starting points. Automate compliance checking and alert on deviations.

Patch Management

Keep all network device firmware and OS software up to date. Many high-profile breaches exploited known, patched vulnerabilities — the patch simply wasn't applied. Establish a patch cadence with emergency procedures for critical vulnerabilities.

Key Hardening Actions for Network Devices

  • 🔌 Disable Telnet — replace with SSH v2 only
  • 🔌 Disable HTTP management interface — use HTTPS only
  • 🔌 Disable SNMPv1 and v2c — upgrade to SNMPv3 with authPriv
  • 🔌 Disable unused switch ports — place in unused VLAN
  • 🔌 Change default credentials on all network devices
  • 🔌 Enable 802.1X on all user-facing switch ports
  • 🔌 Implement DHCP snooping and Dynamic ARP Inspection
  • 🔌 Configure BPDU Guard on access ports (prevent STP manipulation)
  • 🔌 Restrict management access to dedicated OOB network / jump server
  • 🔌 Enable encrypted syslog (TCP syslog over TLS port 6514) to SIEM
  • 🔌 Apply ACLs to restrict management protocols to authorized IPs only
  • 🔌 Implement port security on switches (MAC address limits per port)

SNMPv3 Security Levels

Security Level Authentication Encryption Use Case
noAuthNoPriv ✗ None ✗ None Development/testing only — never use in production. Equivalent security to SNMPv2c.
authNoPriv ✓ MD5 or SHA ✗ None Authenticates the source but data is in plaintext. Better than v1/v2c but not recommended for sensitive data.
authPriv ✓ SHA (required) ✓ AES (required) Use in all production environments. Full authentication and encryption. Use SHA-256 or higher and AES-128 or AES-256.
Exam Key Points — SNMP

SNMPv1 and v2c transmit community strings (essentially passwords) in plaintext — a critical vulnerability. An attacker capturing network traffic can read these strings and use them to read or even modify device configurations (if using the write community string). Always disable SNMP v1/v2c on production devices. If legacy devices require SNMP v1/v2c, isolate them on a management VLAN with strict ACLs and use read-only community strings.

Critical Comparisons

SFTP vs FTPS, RADIUS vs TACACS+, DoH vs DoT vs DNSSEC, TLS 1.2 vs 1.3, EAP-TLS vs PEAP, Split vs Full Tunnel VPN, IDS vs IPS, LDAP vs LDAPS, authPriv vs noAuthNoPriv SNMP

Common Wrong Answers

SFTP is NOT FTP over SSL (that's FTPS). DNSSEC does NOT encrypt DNS traffic. SSH port is 22 NOT 25. TACACS+ runs on TCP NOT UDP. RADIUS only encrypts passwords; TACACS+ encrypts full packet. SNMPv3 still uses ports 161/162.
