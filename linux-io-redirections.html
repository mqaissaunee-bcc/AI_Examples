Understanding File Descriptors

In Linux, everything is a file - including input and output streams. Each process has three standard file descriptors that are automatically opened when the process starts:

File Descriptor Abbreviation Symbol Description
0 STDIN < Standard Input - Data flowing into a program (keyboard input by default)
1 STDOUT > Standard Output - Normal program output (terminal screen by default)
2 STDERR 2> Standard Error - Error messages and diagnostic output (terminal screen by default)

üí° Key Concept: Unless redirected, all three file descriptors point to /dev/tty0 (your terminal). When bash starts, STDIN (fd 0), STDOUT (fd 1), and STDERR (fd 2) are all connected to your terminal, allowing you to type commands and see output.

Simple Redirections

1. Input Redirection (<)

Redirect the contents of a file to serve as input to a command:

# Read input from file instead of keyboard
command < /path/data.txt

# Example: Sort lines from a file
sort < unsorted.txt

# This is equivalent to:
sort unsorted.txt
cat unsorted.txt | sort

2. Output Redirection (>)

Redirect output to a file (overwrites existing content):

# Send output to file (overwrites)
command > output.txt

# Example: List files and save to file
ls -la > directory_listing.txt

# Same as using file descriptor 1 explicitly:
ls -la 1> directory_listing.txt

3. Append Output (>>)

Enhancement: This was missing from the original guide!

# Append output to file (preserves existing content)
command >> output.txt

# Example: Add log entries without destroying previous logs
echo "New log entry" >> application.log
date >> application.log

4. Error Redirection (2>)

Redirect error messages separately from normal output:

# Send errors to file
command 2> errors.txt

# Example: Try to access nonexistent file
cat nonexistent.txt 2> errors.log

# Discard errors (send to /dev/null)
find / -name "*.txt" 2> /dev/null

Example: Output Only

ls -la > list.txt

Changes fd 1 to point to list.txt

Example: Errors Only

grep pattern file.txt 2> errors.txt

Changes fd 2 to point to errors.txt

Example: Input Redirection

wc -l < data.txt

Changes fd 0 to read from data.txt

Advanced Redirections

Redirecting STDOUT and STDERR Together

There are several ways to redirect both output and errors:

# Method 1: Modern syntax (recommended)
command &> output.txt
command >& output.txt

# Method 2: Explicit redirection
command > output.txt 2>&1

# Method 3: Redirect to /dev/null (discard everything)
command > /dev/null 2>&1
‚ö†Ô∏è Important: The order matters! In command > file 2>&1:
  • First, STDOUT (fd 1) is redirected to the file
  • Then, STDERR (fd 2) is redirected to wherever STDOUT is pointing (the file)
  • If you write 2>&1 > file, STDERR goes to the terminal and only STDOUT goes to the file!

Duplicating File Descriptors

# Syntax: [n]>&[m] - Make fd n point to where fd m points

# Example: Duplicate fd 1 to fd 2 (errors go to stdout)
command 2>&1

# Example: Save original stdout, redirect, then restore
exec 3>&1           # Save stdout to fd 3
exec 1>output.txt    # Redirect stdout to file
echo "Goes to file"  
exec 1>&3           # Restore stdout from fd 3
exec 3>&-           # Close fd 3

Separating Output and Errors

# Send output to one file and errors to another
command > output.txt 2> errors.txt

# Example: Compile code, save output and errors separately
gcc program.c > compile_out.txt 2> compile_err.txt

Piping Commands (|)

Enhancement: This essential topic was missing from the original guide!

Pipes connect the STDOUT of one command to the STDIN of another command, creating powerful command chains:

# Basic pipe syntax
command1 | command2

# Example: List files and count them
ls -1 | wc -l

# Example: Search for pattern and sort results
grep "error" logfile.txt | sort | uniq

# Example: Complex pipeline
ps aux | grep python | awk '{print $2}' | head -5
üí° Note: Pipes only connect STDOUT to STDIN. If you want to pipe STDERR as well, use: 
command1 2>&1 | command2

Tee Command - Split Output

Send output to both a file AND the next command in the pipeline:

# Save output to file while passing it along
command1 | tee output.txt | command2

# Example: Save and display simultaneously
ls -la | tee listing.txt | grep "\.txt$"

Here Documents (<<)

Enhancement: Another essential feature not in the original!

Here documents allow you to pass multiple lines of input to a command:

# Here document syntax
command << EOF
line 1
line 2
line 3
EOF

# Example: Create a multi-line file
cat << EOF > config.txt
server=localhost
port=8080
database=mydb
EOF

# Example: Send email with content
mail -s "Subject" user@example.com << EOF
Dear User,
This is the message body.
Regards
EOF

Here Strings (<<<)

# Pass a string as input
command <<< "string content"

# Example: Count characters in a string
wc -c <<< "Hello World"

# Example: Parse variables
read name age <<< "John 30"

Common Patterns and Best Practices

1. Discard Unwanted Output

# Discard all output
command > /dev/null 2>&1

# Keep output, discard errors
command 2> /dev/null

# Keep errors, discard output
command > /dev/null

2. Combine Multiple Commands

# All output to same file
{ command1; command2; command3; } > output.txt 2>&1

# Sequential execution with combined output
(echo "Starting..."; date; ./script.sh) >> log.txt 2>&1

3. Process Substitution

# Compare output of two commands
diff <(ls dir1) <(ls dir2)

# Pass command output as filename
while read line; do echo "$line"; done < <(find . -name "*.txt")
üí° Pro Tips:
  • Always redirect STDERR before piping if you want to include errors
  • Use set -o noclobber to prevent accidental file overwrites with >
  • Use >| to force overwrite when noclobber is set
  • Remember: redirections are processed left to right
  • File descriptors 0-2 are standard; 3-9 are available for custom use
