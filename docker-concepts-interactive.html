What is Docker?

Docker is an open-source platform that automates the deployment, scaling, and management of applications using containerization technology. It allows developers to package applications and their dependencies into lightweight, portable containers that run consistently across any environment.

The "It Works on My Machine" Problem - Solved!

Before Docker, developers faced a common nightmare: applications would work perfectly in development but fail in testing or production. Docker eliminates this by ensuring complete environment consistency.

üìÑ Dockerfile

A Dockerfile is a text file containing a series of instructions that Docker uses to build an image automatically. It's the blueprint for your containerized application.

What is a Dockerfile?

Think of it as a recipe: Just like a recipe lists ingredients and steps to make a dish, a Dockerfile lists the base image, dependencies, and commands needed to create your application environment.

Key Dockerfile Instructions

FROM

Specifies the base image to build upon (e.g., FROM node:14-alpine)

WORKDIR

Sets the working directory for subsequent instructions

COPY / ADD

Copies files from host to container

RUN

Executes commands during build (install packages, etc.)

CMD

Default command to run when container starts

EXPOSE

Declares which ports the container listens on

ENV

Sets environment variables

ENTRYPOINT

Configures container to run as an executable

Example Dockerfile

# Use official Node.js runtime as base image FROM node:14-alpine # Set working directory WORKDIR /app # Copy package files COPY package*.json ./ # Install dependencies RUN npm install # Copy application code COPY . . # Expose port 3000 EXPOSE 3000 # Define environment variable ENV NODE_ENV=production # Run the application CMD ["node", "index.js"]

Best Practices

  • Use official base images from trusted sources
  • Minimize the number of layers by combining RUN commands
  • Use .dockerignore to exclude unnecessary files
  • Order instructions from least to most frequently changing
  • Use multi-stage builds to reduce final image size
  • Don't run containers as root user when possible
  • Keep images small and focused on single purpose

üí° Pro Tip: Multi-Stage Builds

Use multiple FROM statements in a single Dockerfile to create smaller production images. Build and compile in one stage, then copy only the necessary artifacts to the final stage.

üñºÔ∏è Docker Images

A Docker image is a read-only template containing the application code, runtime, libraries, dependencies, and configuration files needed to run an application.

Understanding Images

Class vs Instance Analogy: Think of an image as a class in object-oriented programming and a container as an instance of that class. The image is the blueprint; the container is the running version of that blueprint.

Image Composition

üì¶ Base OS Layer

The foundation - usually a minimal Linux distribution like Alpine, Ubuntu, or Debian

üîß Dependencies

All libraries, frameworks, and system tools your application needs

üíª Application Code

Your actual program files and scripts

‚öôÔ∏è Configuration

Environment variables and configuration files

Image Layers

Docker images are built using a layered filesystem. Each instruction in a Dockerfile creates a new layer:

Layer 4: CMD ["node", "index.js"]
Layer 3: COPY application code
Layer 2: RUN npm install
Layer 1: FROM node:14-alpine (Base)

Each layer is cached, making rebuilds faster when only top layers change

Common Docker Commands for Images

# Build an image from Dockerfile docker build -t myapp:latest . # List all images docker images # Pull an image from registry docker pull nginx:alpine # Tag an image docker tag myapp:latest myusername/myapp:v1.0 # Push image to registry docker push myusername/myapp:v1.0 # Remove an image docker rmi myapp:latest # Inspect image details docker inspect myapp:latest # View image history (layers) docker history myapp:latest

Image Naming Convention

Format: [registry-host]/[username]/[image-name]:[tag]

Example: docker.io/library/nginx:1.21-alpine

  • Registry: docker.io (Docker Hub)
  • Username: library (official images)
  • Image: nginx
  • Tag: 1.21-alpine (version and variant)

Image Optimization Tips

  • Use Alpine Linux base images for smaller sizes (5MB vs 200MB+)
  • Clean up package manager caches in the same RUN command
  • Leverage build cache by ordering instructions strategically
  • Use .dockerignore to exclude unnecessary build context
  • Remove development dependencies in production images
  • Consider using distroless images for maximum security

üì¶ Docker Containers

A container is a runnable instance of an image. It's an isolated environment where your application executes with its own filesystem, networking, and process space.

What Makes Containers Special?

üîí Isolation

Each container runs in its own isolated environment, separate from other containers and the host

‚ö° Lightweight

Share the host OS kernel, making them much lighter than VMs

üöÄ Fast Startup

Start in seconds because they don't boot an entire OS

üîÑ Ephemeral

Designed to be disposable - easily created, destroyed, and replaced

Container Best Practices

  • One process per container (follow the single-responsibility principle)
  • Use health checks to monitor container status
  • Don't store data in containers - use volumes
  • Use restart policies for automatic recovery
  • Limit container resources (CPU, memory) to prevent resource exhaustion
  • Use read-only filesystems when possible for security
  • Implement proper logging and monitoring

üêÑ Cattle, Not Pets

Treat containers as "cattle, not pets" - they should be easily replaceable and identical. Don't manually configure running containers; instead, update the image and redeploy.

Container Communication

DNS-based Discovery

Containers on user-defined bridge networks can reach each other using container names as hostnames:

# Container 'web' can reach 'database' container docker run --network my-app --name web nginx docker run --network my-app --name database postgres # Inside 'web' container, connect to: database:5432

Port Mapping

Expose container ports to the host machine:

# Map host port 8080 to container port 80 docker run -p 8080:80 nginx # Map random host port to container port docker run -P nginx # Bind to specific host interface docker run -p 127.0.0.1:8080:80 nginx # Map UDP port docker run -p 53:53/udp dns-server

Networking Best Practices

  • Use user-defined bridge networks instead of default bridge
  • Create separate networks for different tiers (frontend, backend, database)
  • Use container names for service discovery within networks
  • Expose only necessary ports to the host
  • Implement network policies for security
  • Use overlay networks for multi-host deployments
  • Monitor network traffic and implement rate limiting

üíæ Docker Volumes

Volumes are the preferred mechanism for persisting data generated by and used by Docker containers. Unlike container filesystems, volume data persists even after the container is removed.

Why Volumes Matter

The Problem: When a container is removed, all data inside it is lost. This is fine for stateless applications, but databases, user uploads, and configuration files need to persist.

The Solution: Volumes provide persistent storage that lives outside the container lifecycle.

Types of Mounts

üìÇ Volumes (Recommended)

Managed entirely by Docker. Stored in Docker's directory. Best for persistent data.

Production Ready

üîó Bind Mounts

Mount a specific host directory into container. Direct access to host filesystem.

Development

üí® Tmpfs Mounts

Store data in host memory only. Data lost when container stops. Good for sensitive data.

Temporary

Volume Commands

# Create a named volume docker volume create my-data # List all volumes docker volume ls # Inspect volume details docker volume inspect my-data # Run container with volume docker run -v my-data:/app/data nginx # Run with bind mount docker run -v /host/path:/container/path nginx # Using --mount (verbose syntax) docker run --mount source=my-data,target=/app/data nginx # Remove volume docker volume rm my-data # Remove all unused volumes docker volume prune # Backup volume data docker run --rm -v my-data:/data -v $(pwd):/backup \ alpine tar czf /backup/data-backup.tar.gz /data

Common Use Cases

  • Databases: Store PostgreSQL, MySQL, MongoDB data
  • Configuration: Share config files between containers
  • User Uploads: Persist uploaded files and media
  • Logs: Centralize application logs
  • Development: Hot-reload code changes with bind mounts
  • Backups: Easy to backup and restore volume data

‚ö†Ô∏è Important: Volume Permissions

Volumes inherit permissions from the image. If your app runs as a non-root user, ensure volume permissions match the user ID in the container.

üéº Docker Compose

Docker Compose is a tool for defining and running multi-container Docker applications. With Compose, you use a YAML file to configure your application's services, networks, and volumes.

Why Docker Compose?

Problem: Managing multiple containers with long docker run commands is tedious and error-prone.

Solution: Define everything in a docker-compose.yml file and start your entire application with a single command: docker-compose up

üèóÔ∏è Docker Architecture

Docker uses a client-server architecture. The Docker client talks to the Docker daemon, which does the heavy lifting of building, running, and distributing containers.

Core Components

üñ•Ô∏è Docker Client

Command-line interface (CLI) that users interact with. Sends commands to Docker daemon.

Tools: docker, docker-compose

‚öôÔ∏è Docker Daemon (dockerd)

Background service running on host. Manages Docker objects: images, containers, networks, volumes.

Listens: Docker API requests

üè™ Docker Registry

Stores Docker images. Docker Hub is the public registry, but you can run private registries.

Default: hub.docker.com

Complete Workflow Example

Building and Running a Container:

  1. Developer writes Dockerfile - defines application environment
  2. docker build - client sends request to daemon
  3. Daemon builds image - reads Dockerfile, creates layers
  4. docker push - client uploads image to registry
  5. Production server: docker pull - downloads image from registry
  6. docker run - daemon creates container from image
  7. Container runs - application executes in isolated environment

Communication Flow

# When you type: docker run nginx # Here's what happens: 1. Docker CLI (client) ‚Üí parses command 2. CLI ‚Üí sends HTTP request to Docker daemon via socket 3. Daemon ‚Üí checks if nginx image exists locally 4. If not ‚Üí pulls from Docker Hub registry 5. Daemon ‚Üí creates container from image 6. Daemon ‚Üí allocates network IP, filesystem 7. Daemon ‚Üí starts container process 8. Container ‚Üí runs nginx web server

Key Architecture Concepts

  • Client-Server Model: Separation of CLI and daemon enables remote management
  • Socket Communication: Client and daemon communicate via UNIX socket or network
  • Layered Approach: Images built in layers for efficiency and reusability
  • Copy-on-Write: Containers share image layers, only writing changes
  • Namespaces: Provide isolated workspace for each container
  • Control Groups (cgroups): Limit and monitor resource usage
  • Union Filesystem: Merge multiple layers into single filesystem view

üîí Security Features

Docker leverages Linux kernel features for container isolation:

  • Namespaces: Isolate process trees, networks, users, mounts
  • Control groups: Limit CPU, memory, I/O resources
  • SecComp: Filter system calls
  • AppArmor/SELinux: Mandatory access controls
  • Capabilities: Fine-grained privilege management
