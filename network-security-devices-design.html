Firewalls

Packet Filter • Stateful • NGFW • WAF • Proxy • UTM

What Is a Firewall?

A firewall is a network security control that monitors and filters traffic based on a set of rules. Firewalls operate at various layers of the OSI model — from simple packet-level decisions to sophisticated application-layer inspection. The type of firewall determines what information is available for making allow/deny decisions.

Packet-Filtering Firewall (Layer 3–4)

The simplest firewall type examines each packet independently against a set of ACLACLAccess Control List — a set of rules that specifies permitted or denied actions on a resource, based on source/destination IP, port, and protocol. rules. It inspects source IP, destination IP, source port, destination port, and protocol. It has no memory of previous packets — each packet is evaluated in isolation.

✅ Strengths

  • Very fast — minimal processing overhead
  • Simple to configure and understand
  • Effective for basic access control (block port 23/Telnet)
  • Transparent to end users and applications

⛔ Weaknesses

  • Cannot detect multi-packet attacks (fragmentation)
  • Vulnerable to IP spoofingIP spoofingCrafting packets with a forged source IP address to bypass access controls, impersonate trusted hosts, or obscure the attacker's identity.
  • No application-layer visibility
  • Stateless — cannot track connection state
OSI Layer

Operates at Layer 3 (Network) and Layer 4 (Transport). Decisions based on: Source IP, Destination IP, Protocol (TCP/UDP/ICMP), Source Port, Destination Port.

Stateful Inspection Firewall (Layer 3–4)

Also called a dynamic packet filter, this firewall maintains a state table tracking active connections. It can distinguish between new, established, and related connections — allowing return traffic for established sessions without a separate inbound rule.

State Table Example

When an internal client initiates a TCP connection to an external server (SYN), the firewall records this in the state table. When the server responds (SYN-ACK), the firewall recognizes it as part of an existing outbound session and allows it — without requiring an explicit inbound rule for that source IP.

NEW

First packet of a new connection (e.g., SYN). Evaluated against the full ruleset.

ESTABLISHED

Part of an already-tracked connection. Automatically allowed without re-evaluation.

RELATED

Associated with an existing connection (e.g., FTP data channel). Permitted based on helper modules.

INVALID

Packets that don't match any connection state. Typically dropped.

Next-Generation Firewall (NGFW)

A Next-Generation Firewall integrates traditional stateful inspection with deep application-layer capabilities. NGFWs can identify applications regardless of port (e.g., detect Tor over port 443), inspect encrypted traffic via SSL/TLS inspectionSSL/TLS InspectionA technique where the firewall decrypts, inspects, and re-encrypts SSL/TLS traffic to detect threats hidden in encrypted sessions. Also called SSL interception or TLS break-and-inspect., and enforce policies based on user identity.

SY0-701 Key Point

NGFWs operate at Layer 7 (Application Layer). They can block specific applications (e.g., BitTorrent, Zoom) regardless of the port used, and identify users rather than just IP addresses — critical for policy enforcement in modern enterprises.

Web Application Firewall (WAF)

A Web Application Firewall specifically protects web applications by filtering HTTP/HTTPS traffic between clients and web servers. Unlike NGFWs that identify applications, WAFs understand the content of HTTP transactions and can detect and block web-specific attacks like SQL injectionSQL injectionSQLi — injecting malicious SQL code into input fields to manipulate the backend database, bypassing authentication or extracting data., XSSXSSCross-Site Scripting — an injection attack where malicious scripts are embedded in web pages, allowing session hijacking, credential theft, or malicious redirects., CSRF, and directory traversal.

Positive Security Model

Allow only known-good traffic patterns; deny everything else. More secure but requires more configuration. Used in high-security environments.

Negative Security Model

Block known-bad patterns (signatures); allow everything else. Easier to deploy but misses unknown attacks. Common in most deployments.

WAF Placement

Deployed in front of web servers — either as a hardware appliance, virtual machine, or cloud service (e.g., AWS WAF, Cloudflare). Commonly sits in the DMZDMZDemilitarized Zone — a screened subnet sitting between the internet and the internal network, hosting publicly accessible services while isolating them from internal resources..

WAF vs. NGFW

A WAF protects web applications from Layer 7 web attacks (SQLi, XSS, CSRF). An NGFW provides broad application-layer inspection across many protocols. For a public-facing web app, you want both — the NGFW at the perimeter, the WAF directly in front of web servers.

Proxy Firewall (Application-Layer Gateway)

A proxy firewall acts as an intermediary between clients and servers. Instead of inspecting and forwarding packets, it terminates the connection, inspects the full application-layer content, and initiates a new connection to the destination. This provides maximum inspection capability but at the cost of performance and compatibility.

👤 Forward Proxy

Sits between internal clients and the internet. Clients are configured to route web traffic through the proxy. Provides URL filtering, content inspection, authentication, and caching. Breaks the direct connection between clients and servers.

🌐 Reverse Proxy

Sits in front of internal servers, presenting a single IP to external clients. Provides load balancing, SSL termination, caching, and application-layer protection. Clients are unaware of backend server architecture.

Transparent Proxy

Intercepts traffic without client configuration. Clients are unaware of the proxy. Uses network routing/WCCP to redirect traffic. Common in ISP and enterprise deployments.

Non-Transparent Proxy

Requires explicit client configuration (browser/system proxy settings). Client must send requests to the proxy's address. Easier to enforce authentication.

Unified Threat Management (UTM)

A UTM appliance combines multiple security functions into a single device: firewall, IDS/IPS, antivirus gateway, content filtering, anti-spam, VPN, and web filtering. Designed for simplicity of management, UTMs are common in small-to-medium businesses that lack the staff to manage separate security appliances.

✅ UTM Advantages

  • Single pane of glass management
  • Lower cost than individual appliances
  • Simplified licensing and support
  • Integrated log correlation
  • Vendor coordinates feature integration

⛔ UTM Disadvantages

  • Single point of failure for multiple security functions
  • Performance bottleneck under heavy load
  • Feature depth may lag dedicated appliances
  • "Jack of all trades, master of none"
  • Vendor lock-in risk
Host-Based Firewall

A host-based firewall runs on individual endpoints (Windows Firewall, iptables on Linux, macOS Application Firewall). It filters traffic at the host level, providing protection even within a trusted network segment — critical for Zero Trust and east-west traffic control. Unlike network firewalls, host-based firewalls can apply per-application rules.

Network Segmentation & Zero Trust

DMZ • VLANs • Microsegmentation • Zero Trust • NAC • 802.1X

Why Segment Networks?

Network segmentation divides a network into smaller, isolated zones to contain breaches, limit lateral movement, and enforce access policies. If an attacker compromises one segment, segmentation prevents them from freely moving to other parts of the network. Every major security framework — NIST, CIS, ISO 27001 — mandates network segmentation as a foundational control.

DMZ / Screened Subnet

A DMZDMZDemilitarized Zone — a network segment that sits between the internet and the internal trusted network, providing a buffer zone for publicly accessible services. (also called a screened subnet in modern terminology) is an isolated network segment that hosts services requiring external access — web servers, mail relays, DNS servers — while isolating them from the internal network. Two firewalls create the classic three-zone architecture: Internet Zone, DMZ, and Internal Zone.

VLANs (Virtual Local Area Networks)

VLANs logically segment a physical network at Layer 2, creating separate broadcast domains without requiring physically separate cabling or switches. Devices on different VLANs cannot communicate directly — inter-VLAN routing requires a Layer 3 device (router or Layer 3 switch).

VLAN Use Cases

  • Separate guest Wi-Fi from corporate LAN
  • Isolate IoT/OT devices from user traffic
  • Segment VoIP phones from data traffic
  • Create management VLANs for infrastructure devices

VLAN Hopping

An attack where an adversary exploits switch misconfiguration (DTPDTPDynamic Trunking Protocol — a Cisco proprietary protocol that negotiates trunk links between switches. Attackers can exploit DTP to become a trunk port and access all VLANs.) to gain access to traffic on other VLANs.

Defense: Disable DTP on all non-trunk ports; use a dedicated, unused native VLAN; disable unused ports.

802.1Q Tagging

The IEEE standard for VLAN tagging. Adds a 4-byte VLAN tag (including a 12-bit VLAN ID supporting 4,094 VLANs) to Ethernet frames on trunk ports carrying multiple VLANs.

Microsegmentation

MicrosegmentationMicrosegmentationGranular network segmentation applied at the workload or application level — often using software-defined networking — to enforce least-privilege connectivity between individual servers or containers. takes segmentation to the workload level. Instead of creating large network zones, microsegmentation applies policy at the individual server, VM, or container level. It's the technical implementation that enables Zero Trust east-west traffic control in data centers and cloud environments.

Microsegmentation vs. VLANs

VLANs segment networks at the switch/VLAN level — all servers in a VLAN can communicate with each other freely. Microsegmentation adds controls between individual workloads within the same VLAN — e.g., ensuring a compromised web server cannot reach a database server even if both are on the same VLAN.

Zero Trust Architecture (ZTA)

Zero Trust Architecture (NIST SP 800-207) is the principle that no user, device, or network segment should be implicitly trusted based on location. Every access request — whether from inside or outside the traditional perimeter — must be explicitly authenticated, authorized, and continuously validated before access is granted.

Network Access Control (NAC)

NACNACNetwork Access Control — enforces security policy on network-connecting devices, performing posture assessment and granting access based on compliance with security requirements. controls which devices can connect to a network and what they can access. It performs posture assessment — checking OS version, antivirus status, patch level, and encryption — before granting access. Non-compliant devices may be quarantined to a remediation VLAN.

Evaluates device posture before granting network access. The device must pass a health check to receive full access; non-compliant devices are quarantined.

Intrusion Detection, Prevention & Security Monitoring

IDS • IPS • NIDS • HIDS • SIEM • SOAR • EDR • XDR

IDS vs. IPS: The Core Difference

The fundamental distinction between an Intrusion Detection System (IDS) and an Intrusion Prevention System (IPS) is their response to detected threats. An IDS is a passive sensor — it monitors, logs, and alerts. An IPS is inline — it can drop malicious packets, reset connections, or block IP addresses in real time.

📌 IDS — Intrusion Detection System

  • Passive: Connected via SPAN/mirror port or network tap
  • Monitors a copy of network traffic — does NOT affect traffic flow
  • Generates alerts for security analysts to investigate
  • Cannot block attacks in progress
  • Zero impact on network performance/availability
  • A false positivefalse positiveAn alert generated for activity that is actually benign — a normal event incorrectly classified as malicious. High false positive rates cause alert fatigue. has no operational impact

🛡 IPS — Intrusion Prevention System

  • Inline: All traffic passes through the IPS
  • Can drop, reject, or modify malicious packets in real time
  • Acts as both a detector and a control
  • Adds latency to network traffic
  • A false positive can block legitimate traffic
  • Failure mode is critical — must consider fail-open vs. fail-closedfail-open vs fail-closedFail-open allows all traffic through if the IPS fails (maximizes availability); fail-closed blocks all traffic (maximizes security). The choice depends on the organization's risk tolerance.

Deployment Types: Network vs. Host

NIDS/NIPS — Network-Based

Monitors network traffic for an entire subnet or network segment. Placed at strategic points — typically at the network perimeter and at key internal choke points. Cannot see encrypted traffic (without SSL inspection) and has no visibility into host-level activity.

HIDS/HIPS — Host-Based

Installed as an agent on individual endpoints. Can monitor local system calls, file system changes, registry modifications, and local network connections. Provides visibility inside encrypted tunnels but requires deployment on each host.

Wireless IDS/IPS (WIDS)

Monitors 802.11 wireless traffic for rogue access points, evil twin attacks, deauthentication floods, and unauthorized wireless probing. Uses sensors deployed throughout the wireless environment.

Detection Methods

Signature-Based Detection

Compares network traffic or system activity against a database of known attack patterns (signatures). When traffic matches a signature, an alert is generated or the traffic is blocked. This is the most common detection method and is highly accurate for known attacks.

✅ Advantages

  • Low false positive rate for known attacks
  • Deterministic and explainable results
  • Fast matching using pattern databases
  • Widely understood and well-supported

⛔ Disadvantages

  • Cannot detect unknown (zero-day) attacks
  • Signatures must be continuously updated
  • Attackers can modify code to evade signatures (polymorphism)
  • Does not detect slow, low-and-slow attacks

Anomaly-Based Detection (Statistical)

Establishes a baseline of normal network behavior, then alerts when deviations exceed a threshold. Also called behavior-based or statistical detection. Can detect zero-day attacks because it doesn't rely on known signatures — any sufficiently unusual activity triggers an alert.

False Positive Challenge

Anomaly detection has a higher false positive rate than signature detection because legitimate traffic changes (new applications, seasonal spikes, scanning by IT teams) can appear anomalous. Tuning the baseline and thresholds is an ongoing process.

Baseline Examples

Normal: 500 failed logins/day from internal systems.
Anomaly alert: 50,000 failed logins in 10 minutes — possible brute force attack.
Normal: 2 GB outbound per host per day.
Anomaly alert: 40 GB outbound from a single host — possible data exfiltration.

Heuristic Detection

Uses rules derived from expert knowledge — "this behavior looks like it could be malicious based on general principles." Heuristic rules are more flexible than signatures but less rigorous than statistical baselines. Commonly used in antivirus/endpoint detection to identify variants of known malware families without exact signature matches.

Generic Heuristic Example

"Any process that modifies more than 100 files in 5 seconds and changes their extensions is likely ransomware." — based on ransomware behavior patterns, not specific ransomware signatures.

Machine Learning Enhanced

Modern EDR/XDR platforms use ML models trained on millions of samples. These models learn the characteristics of malicious behavior across many dimensions simultaneously, improving detection of novel threats.

Behavioral / User Entity Behavior Analytics (UEBA)

UEBAUEBAUser and Entity Behavior Analytics — uses machine learning and statistical models to build baseline profiles of normal user and entity behavior, then alerts on significant deviations that may indicate insider threats or compromised accounts. builds profiles of normal behavior for individual users and entities. Unlike simple anomaly detection, UEBA tracks contextual factors: time of day, location, peer group behavior, resource access patterns. Particularly effective for detecting insider threats and compromised accounts that operate within normal traffic volumes but exhibit unusual activity patterns.

SIEM — Security Information and Event Management

A SIEMSIEMSecurity Information and Event Management — aggregates and correlates log data from across the environment to provide centralized threat detection, security alerts, and compliance reporting. platform centralizes collection, aggregation, correlation, and analysis of log data from across the environment. It connects the dots between events that individual systems cannot see in isolation — e.g., correlating a phishing email, a login from an unusual location, and a mass file access event to identify an account compromise.

SOAR — Security Orchestration, Automation & Response

SOARSOARSecurity Orchestration, Automation and Response — automates routine security tasks, orchestrates responses across multiple tools, and provides playbooks for consistent incident handling. extends SIEM by adding automated response. While SIEM generates alerts, SOAR can automatically execute response actions — blocking an IP, quarantining an endpoint, disabling a compromised account — reducing mean time to respond (MTTR).

EDR — Endpoint Detection & Response

Continuous monitoring and collection of endpoint telemetry (processes, network connections, file system activity). Enables real-time threat detection, investigation, and response at the host level. Modern EDR platforms include behavioral analytics and threat hunting capabilities.

XDR — Extended Detection & Response

Integrates EDR with network, email, cloud, and identity telemetry into a unified platform. Provides cross-domain threat detection and automated response. Reduces the silos between endpoint, network, and cloud security teams.

VPN & Remote Access Security

IPsec • SSL/TLS VPN • Split Tunneling • Always-On VPN • Jump Server

Why VPN?

A Virtual Private Network creates an encrypted tunnel over an untrusted network (typically the internet), allowing remote users and branch offices to securely access corporate resources. VPNs provide confidentiality, integrity, and authentication for traffic in transit.

IPsec VPN

IPsecIPsecInternet Protocol Security — a standards-based suite of protocols providing authentication and encryption at the IP layer, operating in transport or tunnel mode. is a standards-based suite of protocols that secures IP communications at the network layer. It is the preferred protocol for site-to-site VPNs and is widely used for remote access. IPsec uses two main protocols and two modes:

AH — Authentication Header

Provides authentication and integrity for IP packets but NOT confidentiality (no encryption). Authenticates the entire packet including the IP header. Incompatible with NAT (modifying the IP header breaks the authentication).

Protocol 51

ESP — Encapsulating Security Payload

Provides confidentiality (encryption), authentication, and integrity. Encrypts the payload; optionally authenticates. NAT-compatible. ESP is the most commonly used IPsec protocol in practice.

Protocol 50

IKE — Internet Key Exchange

Handles the establishment and management of security associations (SAs) and key negotiation. IKEv2 is the modern standard — faster than IKEv1, supports MOBIKEMOBIKEMobility and Multihoming — an IKEv2 extension that allows VPN connections to survive IP address changes, enabling seamless VPN use on mobile devices switching between Wi-Fi and cellular. for mobile connectivity, and built-in NAT traversal.

UDP 500/4500

Transport Mode

Encrypts only the payload — the original IP header is preserved. Used for end-to-end encryption between two hosts. The IP header is visible in transit, making it suitable for host-to-host communication within a network.

[Original IP Header] | [ESP Header] | [Encrypted Payload]

Tunnel Mode

Encrypts both the header and payload — the entire original packet is encapsulated inside a new IP packet. Used for site-to-site VPNs and remote access VPNs. Hides internal IP addressing from transit networks.

[New IP Header] | [ESP Header] | [Encrypted: Original IP + Payload]

SSL/TLS VPN

SSL/TLS VPNs use the TLSTLSTransport Layer Security — the cryptographic protocol that secures web traffic and SSL VPNs. Operates at the application layer, accessible through standard web browsers. protocol (port 443) to create encrypted tunnels. They require no special client software for clientless portal access, making them ideal for contractor and BYOD access through a web browser.

Full Tunnel SSL VPN

Routes ALL client traffic through the VPN tunnel, including personal browsing. Provides maximum visibility and control. Requires a VPN client application. Internet traffic exits from the corporate network.

Split Tunneling

Only corporate-bound traffic goes through the VPN; internet traffic goes directly to the internet. Reduces bandwidth load on corporate network. Security risk: corporate traffic and internet traffic traverse different paths — malware on the endpoint can reach the corporate network through the tunnel.

Clientless SSL VPN (Portal)

Browser-based access to specific web applications via an HTTPS portal. No client software installation required. Limited to web-accessible resources. Common for contractor and vendor access.

Split Tunneling Security Risk

With split tunneling enabled, a compromised endpoint can simultaneously tunnel into the corporate network while being connected to an untrusted network. If the endpoint is infected with malware, the VPN tunnel becomes a direct attack vector into the corporate environment. Always-On VPN with full tunnel or Zero Trust Network Access (ZTNA) are more secure alternatives.

Always-On VPN & ZTNA

Always-On VPN

The VPN connection is automatically established when the device starts, before user login. Ensures all traffic is always protected, enables pre-login authentication for domain joins and remote device management. Prevents users from disabling VPN protection.

ZTNA — Zero Trust Network Access

Provides application-level access based on verified identity and device posture — not network-level VPN access. Users can only reach authorized applications, not the broader network. The modern alternative to traditional VPN for remote access in Zero Trust environments.

Jump Server / Bastion Host

A jump serverjump serverA hardened, audited server that serves as the single access point into a secure network zone, requiring all administrative access to pass through it for monitoring and access control. (also called a bastion host or jump box) is a hardened server that provides a single, monitored access point for administrative access to secure network zones. Administrators connect to the jump server first, then connect to internal systems from there. All administrative sessions are logged and auditable.

Jump Server Best Practices
  • Require MFA for all jump server access
  • Log and audit every session — record keystrokes and screen if required by policy
  • Minimal installed software; hardened OS configuration
  • No internet access from the jump server
  • Separate jump servers for different security zones (e.g., production vs. development)
  • Consider Privileged Access Workstations (PAWs) for highest-security environments

Network Monitoring, Analysis & Deception Technologies

Packet Capture • NetFlow • SNMP • Honeypots • Deception Tech

Why Monitor the Network?

You cannot defend what you cannot see. Network monitoring provides the visibility needed to detect attacks, investigate incidents, enforce policy, and ensure performance. The 2024 M-TRENDS report from Google Mandiant found that the median dwell time for attackers on a network (before detection) was 10 days globally — organizations with mature monitoring programs detected breaches in days, while those without comprehensive monitoring took months. Network visibility is not optional; it's foundational.

Packet Capture & Protocol Analysis

Packet capturepacket captureThe process of intercepting and recording network packets for analysis. Full packet capture records the complete content of every packet; metadata capture records headers only. (pcap) records the complete content of network packets. Protocol analyzers like Wireshark allow detailed inspection of captured traffic — reconstructing sessions, decoding protocols, and identifying anomalies.

SPAN Port / Port Mirroring

A switch port mirroring configuration that copies traffic from one or more ports to a monitoring port where an IDS sensor or packet capture device is connected. Does not affect traffic flow.

Network TAP

A passive hardware device inserted inline that copies all traffic to a monitoring device. Unlike SPAN ports, TAPs capture all traffic with no possibility of packet drops under load. Preferred for forensics and high-accuracy capture.

Out-of-Band Management

A dedicated, separate management network for accessing and configuring network devices. Ensures management access remains available even when the production network is compromised or congested. Uses separate OOB switches/console servers.

NetFlow / IPFIX — Traffic Flow Analysis

NetFlowNetFlowA Cisco protocol (standardized as IPFIX) that exports metadata about IP flows — source/destination IP, port, protocol, byte/packet counts — to a collector for traffic analysis and anomaly detection. captures metadata about network conversations — not the content of packets. Each "flow" record contains: source/destination IP, ports, protocol, duration, bytes, and packets transferred. NetFlow analysis can detect port scans, data exfiltration, botnet communication, and unauthorized lateral movement without capturing raw packet content.

NetFlow vs. Full Packet Capture

NetFlow: Metadata only — who talked to whom, when, how much. Low storage requirements; ideal for long-term trend analysis and anomaly detection. Cannot reconstruct the content of communications.

Full Packet Capture: Complete packet content — enables deep forensic analysis and session reconstruction. High storage requirements; typically retained for 24–72 hours for large networks.

SNMP — Simple Network Management Protocol

SNMPSNMPSimple Network Management Protocol — used to monitor and manage network devices. SNMPv3 adds authentication and encryption; v1/v2 use community strings (essentially cleartext passwords) and should not be used. allows monitoring and management of network devices (routers, switches, servers). Agents on devices respond to queries from an SNMP manager and send traps (alerts) when thresholds are exceeded.

SNMP Security Warning

SNMPv1 and SNMPv2c use plaintext community strings (essentially passwords sent unencrypted). These versions should never be used on modern networks. SNMPv3 adds authentication (MD5/SHA) and optional encryption (DES/AES) — always use SNMPv3 with strong authentication and encryption enabled.

Deception Technologies

Deception technologies deploy decoy systems and data to detect attackers, gather intelligence on their tactics, and mislead them while alerting defenders. Unlike most security controls, deception technologies generate very high-confidence alerts — legitimate users rarely interact with honeypots.

Honeypot

A single system designed to appear valuable but contains no real data. Any interaction with a honeypot is suspicious — legitimate users should never touch it. Used to detect reconnaissance, capture attacker tools, and alert on intrusions.

Deception

Honeynet

A network of honeypot systems designed to simulate an entire network environment. More realistic than a single honeypot; allows studying attacker lateral movement and multi-stage attacks within the fake environment.

Deception

Honeyfile / Honeytokens

Fake files, credentials, or database records placed in high-value locations. If accessed or used, an alert is immediately triggered. Honeyfiles can detect insider threats as well as external attackers who have already breached the perimeter.

Deception

DNS Sinkhole

Redirects DNS queries for known malicious domains to a controlled server (the sinkhole) instead of the actual malicious destination. Used to block malware C2 communication and identify infected systems making those queries.

Honeypots must be carefully designed to entrap rather than entice. Legal advice should be obtained before deploying honeypots, particularly those designed to engage external attackers for extended periods. Interactions within honeypots typically cannot be used as evidence if the honeypot is not owned or authorized by the deploying organization.

Proxy Servers & Content Inspection

Web Proxy / URL Filtering

Inspects and filters HTTP/HTTPS traffic, blocking access to malicious or policy-violating websites. Integrates with threat intelligence feeds to block known-malicious domains in near real time. Often performs SSL inspection to examine HTTPS content.

Email Security Gateway

Filters inbound and outbound email for spam, phishing, malware attachments, and data exfiltration. SPF, DKIM, and DMARC validation blocks email spoofing. Sandboxing analyzes attachments in an isolated environment before delivery.

DLP — Data Loss Prevention

Monitors, detects, and blocks unauthorized transmission of sensitive data — whether through email, web upload, USB devices, or print. Requires data classification to identify what constitutes sensitive data worth protecting.

Secure Network Design Principles

Defense in Depth • Least Privilege • Fail-Secure • Load Balancing • Resilience

Defense in Depth

Defense in depthDefense in DepthA layered security strategy that deploys multiple independent security controls so that if one layer is bypassed or fails, others continue to provide protection. applies multiple layers of overlapping security controls so that no single failure can compromise the entire system. The concept comes from military strategy — creating multiple barriers that an attacker must defeat sequentially. Each layer catches attacks that slip through the previous one.

Key Design Principles

Least Privilege

Every user, system, and process should have only the minimum permissions necessary to perform its function. Limits the blast radius of a compromised account or misconfigured system. Applied to users, service accounts, firewall rules, and API permissions.

Fail-Secure (Fail-Closed)

When a security control fails, it should default to a secure (restrictive) state. A firewall that fails should block all traffic. Contrast with fail-open (allow all on failure) which prioritizes availability over security.

Separation of Duties

No single person should have end-to-end control of a sensitive process. For network design: different teams handle firewall rule changes vs. log review. Prevents insider fraud and reduces the impact of a compromised privileged account.

Implicit Deny

Firewall and ACL rule sets should end with a deny-all rule. Anything not explicitly permitted is denied. This is the foundation of a positive security model — only known-good traffic is allowed.

Network Hardening

Disable unused ports/services, change default credentials, use encrypted management protocols (SSH vs Telnet, HTTPS vs HTTP, SNMPv3 vs v1/v2), implement port security, and apply patches to all network devices.

Secure Topologies

Design networks to minimize attack surface. Redundant paths improve availability. Separate management networks prevent production compromises from affecting management access. Avoid flat networks.

Proxy & Content Filtering Design

Explicit Proxy

Clients configured to use the proxy. Provides strongest authentication and logging capabilities. All web requests clearly identified as proxy traffic.

Transparent Proxy (Intercepting)

Traffic redirected to proxy without client knowledge. No client configuration required. Cannot enforce per-user authentication without additional mechanisms.

SSL/TLS Inspection

Required to inspect HTTPS content. Proxy performs man-in-the-middle with a trusted enterprise CA certificate. Enables malware detection in encrypted traffic but has privacy implications. Exclude sensitive categories (banking, medical) from inspection.
