Installation Media Types

Selecting the appropriate installation media is crucial for successful deployments. Each media type serves different use cases and network environments.

The Binary DVD ISO An ISO image is an archive file containing an identical copy of data found on an optical disc. ISO files are commonly used for distributing operating system installation media. is a complete, self-contained installation image that includes all packages needed for offline installation.

Characteristics

  • Contains both BaseOS The core repository containing the minimal set of packages required for a functioning operating system. Provides the foundation upon which all other software is layered. and AppStream A repository containing additional user-space applications, runtime languages, and databases. Supports multiple versions of the same software through modular packaging. repositories
  • Approximately 8-10 GB in size
  • Ideal for environments without network access
  • Can be burned to physical media or mounted as virtual DVD

Use Cases

  • Air-gapped or isolated networks
  • Locations with limited bandwidth
  • Creating local installation repositories

The Boot ISO is a minimal image that boots the installer and retrieves packages from a network repository.

Characteristics

  • Small size (approximately 700 MB)
  • Requires network connectivity to complete installation
  • Always installs the latest packages from repositories
  • Faster to download and distribute

Use Cases

  • Environments with reliable network connectivity
  • When latest packages are required
  • PXE boot environments
  • Large-scale automated deployments
üí°
Tip: The Boot ISO is ideal for Kickstart deployments where packages are pulled from centralized repositories.

QCOW2 QEMU Copy-On-Write version 2 - A disk image format that supports thin provisioning, snapshots, and compression. The native format for KVM/QEMU virtual machines. images are pre-installed virtual machine disk images ready for immediate deployment.

Characteristics

  • Pre-configured system images
  • Supports thin provisioning and snapshots
  • Native format for KVM virtualization
  • No installation process required

Use Cases

  • Cloud and virtualization environments
  • Rapid VM provisioning
  • Development and testing environments
  • Template-based deployments

Manual Installation Process

The graphical Anaconda installer provides an intuitive interface for configuring system settings. The Installation Summary screen serves as the central hub for all configuration options.

Installation Summary Categories

Keyboard Layout

Configure keyboard layouts for the installed system. Multiple layouts can be added and switched using a keyboard shortcut.

Language Support

Select additional languages for the installed system beyond the installation language.

Time & Date

Configure the system timezone and optionally enable NTP Network Time Protocol - A networking protocol for synchronizing clocks between computer systems. Ensures accurate system time by synchronizing with authoritative time servers. for automatic time synchronization.

Tip: NTP requires network connectivity. If installing offline, disable NTP and set the time manually, then enable it post-installation.

Installation Source

Specifies where to obtain software packages. Options include:

  • Local media - Binary DVD or mounted ISO
  • Network URL - HTTP, HTTPS, FTP, or NFS repository
  • Closest mirror - Automatic mirror selection

  • Server with GUI - Full graphical server environment

  • Minimal Install - Bare minimum packages

Installation Destination

Configure storage devices and partitioning:

Network & Hostname

Configure network interfaces and set the system hostname. Network configuration includes:

  • DHCP or static IP addressing

  • Network interface bonding and teaming
  • VLAN configuration

Kickstart Automated Installation

Kickstart enables fully automated, unattended installations by specifying all configuration parameters in a text file. This ensures consistent, repeatable deployments across multiple systems.

Kickstart File Structure

A Kickstart file consists of three main sections:

The commands section contains directives that configure system settings. Key commands include:

Installation Source Commands

# Install from network repository
url --url="http://server/path/to/repo"

# Install from local CD/DVD
cdrom

# Install from hard drive
harddrive --partition=sdb1 --dir=/path/to/tree

# Add additional repository
repo --name="AppStream" --baseurl="http://server/appstream"

Partitioning Commands

# Clear all partitions
clearpart --all --initlabel

# Automatic partitioning
autopart --type=lvm

# Manual partition definitions
part /boot --fstype=xfs --size=1024
part pv.01 --size=1 --grow
volgroup sysvg pv.01
logvol / --vgname=sysvg --name=root --size=10240
logvol swap --vgname=sysvg --name=swap --size=2048

Network Commands

# DHCP configuration
network --bootproto=dhcp --device=eth0 --activate

# Static IP configuration
network --bootproto=static --ip=192.168.1.100 --netmask=255.255.255.0 --gateway=192.168.1.1 --nameserver=192.168.1.10

# Set hostname
network --hostname=server.example.com

Security Commands

# Root password (encrypted)
rootpw --iscrypted $6$hash...

# SELinux mode
selinux --enforcing

# Firewall configuration
firewall --enabled --ssh

# Create user account
user --name=admin --groups=wheel --iscrypted --password=$6$hash...

The packages section specifies which software to install. It begins with %packages and ends with %end.

Package Selection Syntax

%packages
# Install base environment
@^minimal-environment

# Install package groups
@core
@base
@system-tools

# Install individual packages
vim-enhanced
tmux
bash-completion

# Install module stream
@nodejs:18/common

# Exclude packages
-plymouth
-iwl*firmware
%end
‚ÑπÔ∏è
Package Notation:
  • @^environment - Base environment
  • @group - Package group
  • package - Individual package
  • @module:stream/profile - Modular package
  • -package - Exclude package

Scripts allow custom commands to run before or after installation.

Pre-Installation Script

Runs before the installer configures the system. Useful for dynamic configuration.

%pre
#!/bin/bash
# Determine disk for installation
DISK=$(lsblk -dpno NAME | head -1)
echo "ignoredisk --only-use=${DISK}" > /tmp/disk.cfg
%end

Post-Installation Script

Runs after installation completes. Ideal for final configuration.

%post
#!/bin/bash
# Enable automatic updates
systemctl enable --now dnf-automatic.timer

# Configure SSH
sed -i 's/PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config

# Register with subscription manager
subscription-manager register --auto-attach
%end

Complete Kickstart Example

# System installation settings
text
url --url="http://repo.example.com/baseos"
repo --name="AppStream" --baseurl="http://repo.example.com/appstream"

# Locale settings
keyboard --xlayouts='us'
lang en_US.UTF-8
timezone America/New_York --utc

# Network settings
network --bootproto=dhcp --device=eth0 --activate
network --hostname=server.example.com

# Security settings
rootpw --iscrypted $6$rounds=100000$...
selinux --enforcing
firewall --enabled --ssh

# Disk configuration
bootloader --location=mbr
clearpart --all --initlabel
autopart --type=lvm

# First boot action
firstboot --disabled
reboot

%packages
@^minimal-environment
@core
vim-enhanced
bash-completion
%end

%post
systemctl enable sshd
echo "Installation complete" >> /root/install.log
%end

Kickstart Validation

Validate your Kickstart files before deployment using these tools:

# Validate Kickstart syntax
ksvalidator /path/to/kickstart.cfg

# Compare Kickstart versions
ksverdiff --from RHEL8 --to RHEL9

Publishing Kickstart Files

Kickstart files can be accessed during installation from various sources:

  • HTTP/HTTPS server: inst.ks=http://server/path/ks.cfg
  • FTP server: inst.ks=ftp://server/path/ks.cfg
  • NFS share: inst.ks=nfs:server:/path/ks.cfg
  • Local disk: inst.ks=hd:device:/path/ks.cfg
  • USB drive: inst.ks=hd:UUID=xxxx:/ks.cfg

KVM Virtualization

KVM (Kernel-based Virtual Machine) A full virtualization solution built into the Linux kernel since version 2.6.20. KVM turns Linux into a Type-1 (bare-metal) hypervisor, allowing it to run multiple isolated virtual machines with near-native performance. transforms Linux into a Type-1 hypervisor, enabling efficient virtualization with near-native performance.

KVM Architecture

KVM Virtualization Stack
Interactive diagram showing the KVM virtualization layers. Click or press Enter on each layer to learn more about its role. Use Tab to navigate between layers.
VM 1
Guest OS
Virtual machines run complete operating systems with their own kernel, isolated from other VMs and the host.
VM 2
Guest OS
Each VM has dedicated virtual hardware resources including CPU, memory, storage, and network interfaces.
VM 3
Guest OS
VMs can run different operating systems - Linux, Windows, or other supported guests.
QEMU / libvirt
Userspace virtualization tools
QEMU provides hardware emulation while libvirt offers a unified API for VM management. Together they handle VM creation, device emulation, and lifecycle operations.
KVM Kernel Module
Linux kernel virtualization
The KVM module exposes /dev/kvm interface, enabling user-space programs to create VMs, configure virtual CPUs, and handle memory mapping with hardware-assisted virtualization.
Physical Hardware
CPU with VT-x/AMD-V
Hardware virtualization extensions (Intel VT-x or AMD-V) provide CPU-level support for efficient VM execution. The host's physical resources are divided among virtual machines.

Virtualization Requirements

Before creating virtual machines, verify your system supports hardware virtualization:

# Check CPU virtualization support
grep -E 'vmx|svm' /proc/cpuinfo

# Validate host configuration
virt-host-validate
‚ÑπÔ∏è
CPU Flags: vmx indicates Intel VT-x support; svm indicates AMD-V support. At least one must be present.

Installing Virtualization Host

# Install virtualization group
dnf group install "Virtualization Host"

# Install web console for VM management
dnf install cockpit-machines

# Start and enable libvirtd
systemctl enable --now libvirtd

# Enable web console
systemctl enable --now cockpit.socket

Creating Virtual Machines

virt-install \
  --name=webserver \
  --memory=2048 \
  --vcpus=2 \
  --disk size=20,format=qcow2 \
  --os-variant=rhel9.0 \
  --network network=default \
  --location=/path/to/dvd.iso \
  --graphics vnc \
  --extra-args="inst.ks=http://server/ks.cfg"
